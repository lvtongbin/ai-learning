# Harness Engineering

Harness Engineering是指在 AI 系统中，除模型本身外所有决定系统稳定交付能力的组件总和。其核心目标是解决 AI Agent 在真实场景中的执行稳定性问题，确保模型从"能思考"到"能稳定做事"的关键跨越。

> **`Agent = Model + Harness`**
>
> 模型提供智力，Harness 提供约束、工具与反馈，二者结合构成可落地的自主执行单元。

---

## 一、AI 工程的三阶段演进

| 阶段 | 核心问题 | 解决思路 | 技术重点 | 局限性 |
|------|----------|----------|----------|--------|
| Prompt Engineering（提示词工程） | 模型是否听懂指令 | 优化语言表达 | 角色设定、风格约束、示例引导 | 无法弥补知识缺失，不擅长管理动态信息 |
| Context Engineering（上下文工程） | 模型是否获得正确信息 | 优化信息供给 | 检索增强（RAG）、渐进式披露、信息分层 | 无法解决执行过程中的监督与纠偏问题 |
| Harness Engineering（驾驭工程） | 模型能否稳定执行任务 | 优化运行系统 | 执行编排、状态管理、错误恢复 | 初期搭建成本高，需持续迭代优化 |

三者是包含关系：Harness 包含前两者，是更大系统边界的工程化。

## 二、三大核心支柱

### （一）上下文工程（Context Engineering）

**核心目标**：让 AI Agent 知道"应该做什么"，解决信息不全、理解偏差、上下文溢出（Token 利用率超过 40% 后性能下降）的问题。

关键实践：

- **AGENTS.md 设计**：项目根目录放置"AI 专用导航手册"，约 100 行，作为知识"目录"和"索引"，指向结构化的详细文档，实现知识的渐进式披露，避免 AI 迷失在细节中
- **上下文分层**：按优先级划分信息（高优先级：当前任务相关代码片段；中优先级：架构文档摘要；低优先级：按需查询的参考资料），避免信息过载
- **动态管理**：通过滚动摘要、延迟加载、Token 利用率监控，对抗"上下文腐烂"（冗余信息堆积稀释关键指令）

### （二）架构约束（Architectural Constraints）

**核心目标**：让 AI Agent 知道"不能做什么"，用代码强制执行规则，而非依赖提示词"祈祷"AI 遵守，构建不可突破的硬边界。

关键实践：

- **自定义 Linter 规则**：将架构规范编写成可执行代码，违规时 CI 直接报错并阻断合并，同时提供 AI 可识别的修复方案
- **最小权限原则**：工具白名单 + 操作分级授权（读文件无需确认、写文件提示确认、删除文件必须明确授权）
- **沙箱隔离**：危险操作在隔离环境中执行，不影响生产系统，避免不可逆损失

### （三）熵管理（Entropy Management）

**核心目标**：对抗系统"熵增"，避免代码库文档过时、规则冲突、技术债堆积，保持系统长期健康。

关键实践：

- **文档园丁 Agent**：后台定期运行，扫描文档与代码的一致性，发现过期文档自动提交更新 PR
- **定期清理**：上下文蒸馏（保留结论、丢弃过程细节）、技术债清理 Agent、规则冲突检测

## 三、Harness 的六层架构

### （一）工具层（Tooling Layer）

**核心功能**：为 AI Agent 提供可调用的标准化工具链，是 Agent 与外部环境交互的桥梁。

核心要求：精简、标准化、可复用，每个工具需有清晰的输入输出 schema。

常见工具分类：

- **代码工具**：ESLint、Prettier、TS 类型检查、StyleLint（代码规范校验）
- **测试工具**：Jest/Vitest、Cypress（自动测试、结果校验）
- **工程工具**：Git、npm/yarn、Docker、CI/CD 流水线
- **辅助工具**：文件读写、浏览器自动化、API 调用、日志查询（LogQL）、指标监控（PromQL）

### （二）记忆层（Memory Layer）

**核心功能**：实现 Agent 持久的记忆和知识存储，解决 AI 短期记忆不足的问题。

核心实践：有价值，采用 AGENTS.md 导航地图 + 结构化 `docs/` 目录的模式，AGENTS.md 作为"索引"，`docs/` 目录存储详细知识，同时将文档纳入版本控制和自动化校验。

### （三）上下文管理（Context Management）

**核心功能**：专门对抗"Context Rot（上下文腐烂）"，确保上下文窗口的高信噪比。

关键策略：历史上下文摘要压缩、工具输出卸载（上下文仅保留摘要和文件引用）、任务阶段动态加载/卸载相关技能定义。

### （四）任务编排（Task Orchestration）

**核心功能**：负责工作流管理和状态机，实现"Plan-Act-Observe-Reflect"闭环。

典型流程：

```
目标理解 → 信息检查 → 分析处理 → 输出生成 → 结果检查 → 修正迭代
```

核心能力：任务断点续传、超时重试、异常回滚、多 Agent 协调（Manager-Workers 架构）、结果汇总与校验。

### （五）验证过滤（Validation & Filtering）

**核心功能**：自动化的质量检测机制，是 Harness 系统的"质量关卡"。

三层验证体系：

| 层级 | 类型 | 内容 | 失败处理 |
|------|------|------|----------|
| L1 | 硬性规则 | 格式、语法校验（Prettier、ESLint） | 直接阻断 |
| L2 | 执行测试 | 在隔离沙箱中运行代码，验证功能正确性 | 触发修复流程 |
| L3 | 软性标准 | 调用高推理模型进行"同行评审"，校验逻辑与架构合规性 | 反馈修正建议 |

> 关键要点：校验失败产生的报错信息需"写给 Agent 看"，并自带修复指令，形成无需人工介入的修正闭环。

### （六）自我修正（Self-Correction）

**核心功能**：反馈闭环和错误修复能力，实现"一次犯错、永久杜绝"。

核心流程：

```
捕获错误 → 提取错误类型与根源 → 引导 Agent 自主反思与修正 → 沉淀为新校验规则 → 优化 Harness 系统
```

---

## 四、核心工作流程

Harness Engineering 遵循"人类掌舵、Agent 执行、Harness 管控、持续迭代"的闭环：

1. **人类设计 Harness**：定义项目架构、编码规范、约束规则、工具集、校验体系，明确 Agent 的工作边界与目标
2. **Agent 自主执行**：基于 Harness 提供的上下文、工具与规则，自主理解需求、拆分任务、调用工具、生成代码、执行自测、提交 PR
3. **Harness 自动校验**：验证过滤组件启动三层校验，违规则触发自我修正流程
4. **人类极简审核**：工程师无需关注代码细节，仅审核核心业务逻辑与架构合理性，实现"少干预、高管控"
5. **反馈迭代 Harness**：收集错误与问题，优化约束规则、工具集与上下文管理策略，将高频错误沉淀为新规则

## 五、一线公司实践案例

### Anthropic 的自主编码系统

**核心问题 1：长任务上下文过载**

- 解决方案：Context Reset（上下文重置）
- 类比：内存泄漏后重启进程，而非清理缓存

**核心问题 2：自评失真**

- 解决方案：角色分离架构

| 角色 | 职责 |
|------|------|
| Planner（规划者） | 需求转规格 |
| Generator（生成器） | 逐步实现 |
| Evaluator（评估者） | 环境化测试（真实操作验证） |

### OpenAI 百万行零手写代码实验

**背景**：2026 年 2 月，3 人工程师团队，5 个月内从零构建百万行代码的内部产品，全程零人工手写代码，效率达到传统开发的 10 倍。

**核心 Harness 实践**：

- **AGENTS.md 范式**：早期千页级文档导致 Agent 迷失细节；后调整为 100 行导航地图，实现知识渐进式披露，文档纳入版本控制与自动化校验
- **全链路可观测性**：接入 Chrome DevTools 协议，让 Agent 可直接读取 UI、截取 DOM 快照；搭建本地可观测性栈，支持 Agent 用 LogQL 查询日志、PromQL 查询指标
- **严格约束与校验**：自定义 Linter 规则强制执行架构分层，三层验证体系确保代码质量，高频错误自动沉淀为新规则
- **工程师角色转变**：80% 时间用于构建 Harness，核心工作是设计环境、明确意图、维护反馈回路

**核心启示**：模型是商品，Harness 是护城河——当基础模型能力越过临界点后，围绕模型构建的 Harness 系统质量，成为决定产品成败的关键。

### LangChain 推理三明治策略

**背景**：通过引入"推理三明治"策略，在不更换模型的情况下，代码正确率从 52.8% 提升至 66.5%，行业排名从第 30 跃升至第 5。

**推理三明治策略**：

| 阶段 | 推理模式 | 目标 |
|------|----------|------|
| 规划阶段 | 高推理 | 仔细分析需求，制定详细执行计划 |
| 执行阶段 | 低推理 | 按计划生成代码，节省算力 |
| 验证阶段 | 高推理 | 严格审查代码，结合测试结果自我修正 |

## 六、常见陷阱与最佳实践

### 常见陷阱

| 陷阱 | 问题 | 建议 |
|------|------|------|
| 过度工程化 | 初期搭建复杂系统，维护困难，反而降低效率 | 循序渐进，优先满足核心管控需求 |
| 静态 Harness 设计 | 系统一成不变，约束逐渐成为"阻碍" | 建立迭代机制，定期优化规则与工具集 |
| 忽视文档基础设施 | 未维护 AGENTS.md，Agent 频繁犯错 | 安排专人或文档园丁 Agent 维护文档一致性 |
| 缺乏反馈循环 | 仅有约束与校验，Agent 重复犯同一错误 | 确保校验失败后 Agent 能收到明确修复指令，高频错误沉淀为新规则 |

### 最佳实践

- **约束规则需"可执行"**：避免模糊规则（如"代码要规范"），需转化为可校验的代码或脚本
- **工具封装需标准化**：每个工具的输入输出必须清晰，避免 Agent 无法理解如何调用
- **沙箱隔离必启用**：危险操作（如删除文件、修改生产代码）需在 Docker 沙箱中执行
- **先基础后高级**：先搭建基础 Harness（约束+校验+工具），再逐步添加记忆层、熵管理等高级功能

## 七、关键洞察

- **能力边界**：Prompt 解决"说清楚"，Context 解决"信息对"，Harness 解决"持续做对"
- **包含关系**：Harness 包含前两者，是更大系统边界的工程化
- **落地关键**：模型决定上限，Harness 决定能否稳定落地
- **发展趋势**：AI 落地挑战正从"让模型聪明"转向"让模型稳定工作"
- **工程师价值**：从"代码实现"提升为"系统设计与意图定义"
